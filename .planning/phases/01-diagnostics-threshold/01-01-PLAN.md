<objective>
Make Python parse failures visible (DIAG-01). Today `FastAPIParser.extract_endpoints` silently swallows `SyntaxError`/`OSError` at `fastapi_parser.py:382` (`except (SyntaxError, OSError): continue`), so a malformed source file vanishes with no signal. This plan accumulates the skipped filepath on the parser instance, writes a named WARNING to stderr, and threads the skipped-file list onto `DriftReportMetadata.to_dict()` so it appears in JSON output.

Purpose: A developer whose source file fails to parse must see which file was skipped instead of getting a falsely-clean run. This is the first of the three diagnostics fixes and is the prerequisite signal that everything else (zero-endpoint detection in plan 02) relies on.

Output: `FastAPIParser.__init__` initializing `self.source_warnings`, stderr warning on parse failure, a `source_warnings` field on `DriftReportMetadata` with matching `to_dict()` key, and three new parser tests.
</objective>

## Phase Goal

**As a** developer running DocGuard in CI, **I want to** see a named warning whenever a source file fails to parse, **so that** a malformed file is never silently dropped and reported as a clean run.
